## Context

当前 MystiSql 的 TUI 使用 Bubble Tea 框架实现，这是一个功能强大的 TUI 框架，但它采用全屏渲染模式和复杂的组件模型。这种设计适合构建复杂的终端应用（如文件管理器、系统监控工具），但对于 SQL 客户端这类 REPL 风格的工具来说过于复杂。

**当前问题**:
1. 全屏渲染导致输出无法滚动查看历史记录
2. 复杂的窗口布局与 MySQL CLI 体验差异大
3. Bubble Tea 的事件驱动模型增加了不必要的复杂度
4. 用户需要学习新的快捷键和交互方式

**目标用户**: 熟悉 MySQL/PostgreSQL CLI 的数据库管理员和开发者

## Goals / Non-Goals

**Goals:**
- 实现与 MySQL CLI 高度相似的交互体验
- 支持真正的多行 SQL 输入（以分号结尾）
- 保持流式输出，允许用户滚动查看历史
- 保留核心功能：SQL 执行、实例切换、历史命令、结果导出
- 减少外部依赖，使用标准库实现

**Non-Goals:**
- 不实现复杂的全屏 TUI 功能（如分页、多窗口）
- 不支持鼠标交互
- 不实现高级编辑功能（如复制粘贴、撤销重做）
- 不改变现有的查询引擎和连接管理逻辑

## Decisions

### Decision 1: 使用标准库实现 REPL 而非 TUI 框架

**选择**: 使用 `bufio.Scanner` + `golang.org/x/term` 实现 REPL 风格界面

**理由**:
- REPL 模式更接近 MySQL CLI 的用户体验
- 标准库实现更轻量，无额外依赖
- 流式输出更自然，支持终端原生滚动
- 代码更简单，易于维护

**替代方案**:
- 继续使用 Bubble Tea 但简化布局：仍然无法解决全屏渲染的根本问题
- 使用其他 TUI 框架（如 tview）：同样存在全屏渲染问题
- 使用 GNU Readline 绑定：增加 CGO 依赖，跨平台兼容性差

### Decision 2: 多行输入处理策略

**选择**: 使用分号作为 SQL 语句结束符，支持跨行输入

**实现方式**:
```
mystisql@local-mysql> SELECT *
    -> FROM users
    -> WHERE id = 1;
```

**理由**:
- 与 MySQL CLI 行为一致
- 用户无需学习新的输入方式
- 支持复杂的 SQL 语句格式化

**状态提示**:
- `mystisql@instance>` - 新语句开始
- `    ->` - 语句继续
- `    '>` - 单引号字符串内
- `    ">` - 双引号字符串内

### Decision 3: 输出格式设计

**选择**: 模仿 MySQL CLI 的表格输出格式

**示例输出**:
```
+----+-------+------------------+
| id | name  | email            |
+----+-------+------------------+
|  1 | Alice | alice@test.com   |
|  2 | Bob   | bob@test.com     |
+----+-------+------------------+
2 rows in set (0.005 sec)
```

**理由**:
- 用户熟悉的格式
- 自动计算列宽
- 支持 NULL 值显示为 NULL

### Decision 4: 实例切换方式

**选择**: 支持 `USE <instance>` 命令和命令行参数

**实现方式**:
```sql
-- 切换实例
mystisql@local-mysql> USE local-postgres;
Database changed

-- 显示当前实例
mystisql@local-mysql> SELECT INSTANCE();
+----------------+
| INSTANCE()     |
+----------------+
| local-mysql    |
+----------------+

-- 显示所有实例
mystisql@local-mysql> SHOW INSTANCES;
+----------------+----------+--------+
| name           | type     | status |
+----------------+----------+--------+
| local-mysql    | mysql    | online |
| local-postgres | postgres | online |
+----------------+----------+--------+
```

**理由**:
- 与 SQL 语法风格一致
- 无需记忆额外快捷键
- 支持脚本自动化

### Decision 5: 历史命令存储

**选择**: 使用文件存储历史命令，路径 `~/.mystisql/history`

**格式**:
```
SELECT * FROM users LIMIT 10;
SHOW DATABASES;
USE local-postgres;
```

**理由**:
- 与 MySQL CLI 的 `.mysql_history` 行为一致
- 简单的文本格式，易于调试
- 支持上下箭头浏览历史

### Decision 6: 终端输出换行符处理

**选择**: 使用 `\r\n` 作为换行符，而非单独的 `\n`

**理由**:
- **跨平台兼容性**: Windows 终端需要 `\r\n`，Unix/Linux 终端也能正确处理
- **光标定位准确**: 确保每行输出后光标正确定位到行首
- **表格对齐**: 避免因换行符不一致导致的表格列错位问题
- **终端原始模式**: 在原始模式下，`\n` 只会移动光标到下一行，不会回到行首

**实现要点**:
```go
// 正确的换行输出
fmt.Print("line 1\r\nline 2\r\n")

// 表格输出示例
fmt.Print("+----+-------+\r\n")
fmt.Print("| id | name  |\r\n")
fmt.Print("+----+-------+\r\n")
```

**注意事项**:
- 使用 `fmt.Print()` 或 `fmt.Fprintf()` 时手动添加 `\r\n`
- 避免使用 `fmt.Println()`，因为它在不同平台可能产生不同结果
- 在写入文件时可以继续使用 `\n`（文件不需要 `\r\n`）

### Decision 7: 架构设计

**核心组件**:

```
┌─────────────────────────────────────────────────────────┐
│                      REPL Engine                         │
├─────────────────────────────────────────────────────────┤
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐   │
│  │   Input      │  │   Parser     │  │   Executor   │   │
│  │   Handler    │──│   (SQL)      │──│   (Query)    │   │
│  └──────────────┘  └──────────────┘  └──────────────┘   │
│         │                                     │         │
│         ▼                                     ▼         │
│  ┌──────────────┐                    ┌──────────────┐   │
│  │   History    │                    │   Formatter  │   │
│  │   Manager    │                    │   (Output)   │   │
│  └──────────────┘                    └──────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**模块职责**:
- `InputHandler`: 处理用户输入，支持多行模式
- `SQLParser`: 解析 SQL，识别语句边界
- `Executor`: 调用 QueryEngine 执行 SQL
- `Formatter`: 格式化输出结果
- `HistoryManager`: 管理命令历史

## Risks / Trade-offs

### Risk 1: 功能简化可能影响部分用户

**风险**: 移除全屏 TUI 后，部分用户可能失去他们习惯的功能（如分页、搜索）

**缓解措施**:
- 保留结果导出功能（CSV/JSON）
- 使用终端原生的滚动和搜索功能（用户可使用 `less` 管道）
- 在文档中说明如何使用外部工具增强体验

### Risk 2: 多行输入解析复杂度

**风险**: 正确处理 SQL 中的字符串、注释等可能导致解析错误

**缓解措施**:
- 使用成熟的 SQL 解析器（如现有的 SQL validator）
- 编写充分的单元测试覆盖边界情况
- 提供清晰的错误提示

### Risk 3: 终端兼容性

**风险**: 不同终端对原始模式的支持可能有差异

**缓解措施**:
- 使用 `golang.org/x/term` 库，已处理大部分兼容性问题
- 提供降级模式（非原始模式）
- 在 CI 中测试多种终端环境

### Trade-off: 功能 vs 简洁

**取舍**: 放弃复杂的 TUI 功能换取更接近 MySQL CLI 的体验

**理由**: MystiSql 的核心价值是作为数据库访问网关，而非终端工具。用户更看重与现有工具的兼容性，而非独特的交互方式。

## Migration Plan

### Phase 1: 并行开发（不影响现有功能）

1. 创建新的 REPL 实现在 `internal/cli/repl/` 目录
2. 保留现有 `tui.go` 不变
3. 通过配置或命令行参数选择使用哪种模式

### Phase 2: 切换默认模式

1. 将 REPL 模式设为默认
2. 保留 `--tui` 参数用于回退到旧模式
3. 收集用户反馈

### Phase 3: 清理

1. 移除 Bubble Tea 依赖
2. 删除旧 TUI 代码
3. 更新文档

### Rollback Strategy

如果新 REPL 模式出现严重问题：
1. 通过配置项快速切换回旧 TUI 模式
2. 发布修复版本
3. 代码保留在独立分支，可快速恢复

## Open Questions

1. **是否需要支持 `\G` 垂直输出格式？**
   - MySQL CLI 支持 `SELECT * FROM users\G` 垂直显示
   - 建议：初期不支持，根据用户反馈决定

2. **是否需要支持自定义提示符？**
   - MySQL 支持 `prompt` 命令自定义提示符
   - 建议：初期使用固定格式，后续可扩展

3. **是否需要支持 `\.` 执行脚本文件？**
   - MySQL 支持 `\. /path/to/script.sql` 执行脚本
   - 建议：支持 `source` 命令作为替代
