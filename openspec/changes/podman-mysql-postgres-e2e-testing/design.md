## Context

MystiSql 是一个数据库访问网关，支持 MySQL、PostgreSQL、Oracle 和 Redis。目前项目已完成 Phase 3 的核心功能开发（认证、审计、SQL 验证），但缺少真实的端到端测试环境来验证这些功能。开发团队需要一个可重复、可移植的测试环境，用于在真实数据库上执行 e2e 测试。

当前状态：
- 已有单元测试和集成测试，但缺少真实数据库环境的 e2e 测试
- 开发环境不统一，部分开发者使用 Docker，部分使用本地安装的数据库
- 测试数据管理分散，缺乏统一的初始化和清理机制

约束条件：
- 必须使用容器化技术（Podman）以确保环境一致性
- 测试环境启动和停止必须快速（< 30秒）
- 支持并发运行多个测试实例（避免端口冲突）
- 测试数据必须可重置，不依赖外部状态

## Goals / Non-Goals

**Goals:**

- 提供一键启动/停止的测试数据库环境（MySQL 8 + PostgreSQL 14+）
- 自动化测试数据库的初始化（建库、建表、插入测试数据）
- 提供完整的 e2e 测试用例覆盖核心功能：
  - MySQL 连接池：连接、查询、重连、健康检查
  - PostgreSQL 驱动：连接、查询、SSL 模式、超时控制
  - 事务管理：BEGIN、COMMIT、ROLLBACK、并发事务
  - 批量操作：批量插入、批量更新、错误处理
  - Phase 3 功能：认证中间件、审计日志、SQL 验证
- 测试环境支持 CI/CD 流程集成
- 提供清晰的测试报告和失败诊断信息

**Non-Goals:**

- 不替代单元测试和集成测试
- 不测试 Oracle 和 Redis（当前阶段专注于 MySQL 和 PostgreSQL）
- 不测试性能和压力测试（未来可作为独立任务）
- 不实现生产级数据库配置（仅用于测试）

## Decisions

### 1. 容器技术选择：Podman vs Docker

**决定**: 使用 Podman

**理由**:
- Podman 是无守护进程（daemonless）架构，更安全
- 支持 rootless 模式，无需 root 权限
- 与 Docker CLI 兼容，学习曲线低
- 在 Linux 环境下性能更好（无需虚拟化层）

**替代方案**:
- Docker: 需要守护进程，权限管理更复杂
- docker-compose: 可行，但本项目使用 Podman 更轻量

### 2. 测试数据库版本选择

**决定**: MySQL 8.0, PostgreSQL 14+

**理由**:
- MySQL 8.0 是当前主流版本，支持 CTE、窗口函数等新特性
- PostgreSQL 14+ 性能优异，支持更好的并行查询
- 与生产环境版本保持一致

### 3. 测试数据初始化策略

**决定**: 使用 SQL 初始化脚本 + 测试前清理

**理由**:
- SQL 脚本简单直观，易于维护
- 测试前清理确保测试数据一致性
- 可以通过容器 volume mount 挂载初始化脚本

**替代方案**:
- 代码中动态建表：灵活性高但测试代码冗余
- 使用数据库 migration 工具：过度设计，增加复杂度

### 4. 测试框架选择

**决定**: Go 标准测试框架 + testify

**理由**:
- 项目已使用 Go 标准测试框架
- testify 提供更好的断言和 mock 功能
- 无需引入额外测试框架

### 5. 测试隔离策略

**决定**: 每个测试用例使用独立的数据库 schema/database

**理由**:
- 避免测试用例之间相互干扰
- 支持并发运行测试用例
- 测试失败时更容易定位问题

**替代方案**:
- 共享数据库：可能导致测试不稳定
- 每个测试一个容器：启动时间太长

### 6. 配置管理

**决定**: 使用单独的 e2e 测试配置文件 `config/e2e-test.yaml`

**理由**:
- 与开发/生产配置分离
- 可以自定义测试数据库连接参数
- 支持 CI/CD 环境变量覆盖

## Risks / Trade-offs

### 风险 1: 容器启动失败
- **风险**: Podman 未安装或权限不足导致测试环境无法启动
- **缓解**: 
  - 提供环境检查脚本，启动前验证 Podman 是否可用
  - 文档中明确说明前置条件
  - 提供 fallback 方案（使用环境变量配置外部数据库）

### 风险 2: 测试执行时间过长
- **风险**: e2e 测试启动容器和初始化数据库耗时较长
- **缓解**: 
  - 使用轻量级数据库镜像
  - 测试数据最小化（仅包含必要数据）
  - 支持跳过 e2e 测试（通过构建标签）

### 风险 3: 测试不稳定
- **风险**: 网络延迟、数据库启动慢导致测试偶尔失败
- **缓解**: 
  - 添加重试机制和等待逻辑
  - 使用健康检查确保数据库就绪后再执行测试
  - 设置合理的超时时间

### 风险 4: 端口冲突
- **风险**: 默认端口（3306, 5432）已被占用
- **缓解**: 
  - 使用随机端口或可配置端口
  - 测试启动前检查端口可用性
  - 文档中说明如何修改端口配置

### 权衡 1: 测试覆盖度 vs 执行时间
- **权衡**: 更全面的测试覆盖 vs 更快的测试执行
- **选择**: 优先保证核心功能的测试覆盖，非关键路径可选择性测试
- **理由**: 平衡测试质量和开发效率

### 权衡 2: 环境真实性 vs 资源消耗
- **权衡**: 使用完整生产配置 vs 使用最小化测试配置
- **选择**: 使用最小化测试配置
- **理由**: 降低资源消耗，加快测试速度，核心功能仍能充分验证
