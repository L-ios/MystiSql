# SQL 执行 TUI 功能规范

## Purpose

定义 MystiSql TUI 中 SQL 执行功能的规范，特别是针对长期运行的 SQL 查询的处理，确保用户可以在 TUI 界面中流畅地执行和管理 SQL 查询。

## Requirements

### Requirement: 基本 SQL 执行

系统必须支持基本的 SQL 查询执行功能。

#### Scenario: 异步执行 SQL

- **WHEN** 用户提交 SQL 查询
- **THEN** 系统必须异步执行该查询
- **AND** TUI 界面必须保持响应不阻塞
- **AND** 用户必须能够继续输入其他命令或操作

#### Scenario: 结果表格显示

- **WHEN** 查询执行完成
- **THEN** 系统必须以表格形式显示查询结果
- **AND** 表格必须包含列标题
- **AND** 表格必须支持自动调整列宽
- **AND** 表格必须支持水平滚动

#### Scenario: 结果分页

- **WHEN** 查询返回大量数据
- **THEN** 系统必须支持结果分页
- **AND** 必须显示当前页码和总页数
- **AND** 用户可以使用快捷键翻页（如 Page Up/Down）
- **AND** 每页默认显示 50-100 行

#### Scenario: 结果导出

- **WHEN** 用户选择导出查询结果
- **THEN** 系统必须支持导出为 CSV、JSON 等格式
- **AND** 必须提示用户选择保存位置
- **AND** 导出过程必须在后台进行不阻塞界面

---

### Requirement: 长期运行查询管理

系统必须支持管理长期运行的 SQL 查询。

#### Scenario: 显示查询进度

- **WHEN** 查询正在执行中
- **THEN** 系统必须显示查询的实时进度
- **AND** 必须显示已执行时间
- **AND** 必须显示状态指示器（如旋转图标、进度条）

#### Scenario: 显示查询状态

- **WHEN** 查询正在执行
- **THEN** 状态栏必须显示当前查询状态
- **AND** 状态必须包括：执行中、完成、失败
- **AND** 不同状态必须用不同颜色表示

#### Scenario: 取消正在执行的查询

- **WHEN** 用户按取消键（如 Ctrl+C）或点击取消按钮
- **THEN** 系统必须取消当前正在执行的查询
- **AND** 必须在状态栏显示"查询已取消"消息
- **AND** 数据库连接必须保持可用

#### Scenario: 显示执行时间

- **WHEN** 查询执行完成
- **THEN** 系统必须显示总执行时间
- **AND** 时间格式必须精确到毫秒
- **AND** 必须显示在结果区域的底部或状态栏

---

### Requirement: 查询结果处理

系统必须支持对查询结果进行处理和分析。

#### Scenario: 结果排序

- **WHEN** 用户点击列标题或使用排序快捷键
- **THEN** 系统必须按该列对结果进行排序
- **AND** 必须支持升序和降序切换
- **AND** 排序必须在客户端进行（不重新查询）

#### Scenario: 结果过滤

- **WHEN** 用户输入过滤条件
- **THEN** 系统必须过滤显示符合条件的结果行
- **AND** 必须支持简单的文本匹配过滤
- **AND** 过滤必须实时生效

#### Scenario: 结果搜索

- **WHEN** 用户使用搜索功能（如 Ctrl+F）
- **THEN** 系统必须在结果中高亮显示匹配的值
- **AND** 必须支持跳转到下一个/上一个匹配项
- **AND** 必须显示匹配数量

#### Scenario: 显示结果统计

- **WHEN** 查询结果加载完成
- **THEN** 系统必须显示统计信息
- **AND** 必须包括：总行数、总列数
- **AND** 必须显示在状态栏或结果区域底部

---

### Requirement: 批量执行

系统必须支持批量执行多条 SQL 命令。

#### Scenario: 批量提交

- **WHEN** 用户输入多条 SQL 命令（用分号分隔）
- **THEN** 系统必须识别为批量执行
- **AND** 必须按顺序执行每条命令
- **AND** 必须显示批量执行进度（如 "2/5"）

#### Scenario: 执行顺序控制

- **WHEN** 批量执行多条命令
- **THEN** 系统必须严格按照用户输入的顺序执行
- **AND** 必须等待前一条命令完成再执行下一条
- **AND** 必须显示当前正在执行的命令

#### Scenario: 批量执行错误处理

- **WHEN** 批量执行过程中某条命令失败
- **THEN** 系统必须提示用户选择：停止执行或继续执行
- **AND** 如果选择继续，必须标记失败的命令
- **AND** 必须在最后显示执行摘要（成功数/失败数）

---

### Requirement: 事务支持

系统必须在 TUI 中支持事务管理。

#### Scenario: 开始事务

- **WHEN** 用户执行 BEGIN TRANSACTION 命令
- **THEN** 系统必须在当前连接上开始事务
- **AND** 状态栏必须显示"事务进行中"状态
- **AND** 必须记录事务开始时间

#### Scenario: 提交事务

- **WHEN** 用户执行 COMMIT 命令
- **THEN** 系统必须提交当前事务
- **AND** 必须在状态栏显示"事务已提交"消息
- **AND** 必须更新事务状态为"无事务"

#### Scenario: 回滚事务

- **WHEN** 用户执行 ROLLBACK 命令
- **THEN** 系统必须回滚当前事务
- **AND** 必须在状态栏显示"事务已回滚"消息
- **AND** 必须更新事务状态为"无事务"

#### Scenario: 显示事务状态

- **WHEN** 事务正在进行中
- **THEN** 状态栏必须显示当前事务状态
- **AND** 必须显示事务已运行的时间
- **AND** 必须提示用户可以 COMMIT 或 ROLLBACK

#### Scenario: 设置事务隔离级别

- **WHEN** 用户设置事务隔离级别（如 SET TRANSACTION ISOLATION LEVEL）
- **THEN** 系统必须应用新的隔离级别
- **AND** 必须在状态栏显示当前隔离级别
- **AND** 必须在下一条 SQL 命令执行时生效

---

### Requirement: 性能优化

系统必须优化查询性能和用户体验。

#### Scenario: 查询结果缓存

- **WHEN** 用户执行相同的查询
- **THEN** 系统可以返回缓存的结果（如果缓存未过期）
- **AND** 缓存必须有合理的过期时间（如 5 分钟）
- **AND** 用户必须能够手动刷新缓存

#### Scenario: 连接池利用

- **WHEN** 执行 SQL 查询
- **THEN** 系统必须从连接池获取连接
- **AND** 查询完成后必须归还连接到连接池
- **AND** 必须显示连接池状态（如"连接 3/10"）

#### Scenario: 显示查询执行计划

- **WHEN** 用户请求查看执行计划（如 EXPLAIN 命令）
- **THEN** 系统必须显示查询的执行计划
- **AND** 必须以可读格式展示（如表格或树形结构）
- **AND** 必须包括：扫描类型、索引使用、预估行数等信息

---

### Requirement: 错误处理和恢复

系统必须正确处理 SQL 执行错误。

#### Scenario: SQL 语法错误

- **WHEN** 用户执行的 SQL 有语法错误
- **THEN** 系统必须显示数据库返回的错误信息
- **AND** 必须高亮显示错误位置（如果可用）
- **AND** 必须允许用户修改并重新执行

#### Scenario: 连接错误

- **WHEN** 执行 SQL 时连接断开
- **THEN** 系统必须显示连接错误信息
- **AND** 必须自动触发重连机制
- **AND** 重连成功后必须提示用户重新执行查询

#### Scenario: 超时错误

- **WHEN** 查询执行超过配置的超时时间
- **THEN** 系统必须自动取消查询
- **AND** 必须显示"查询超时"错误信息
- **AND** 必须建议用户优化查询或增加超时时间
